ADR 0001 — Setzer Architecture Accepted

Status: Accepted (design) · Date: 2026-06-01 · Project: Setzer (standalone tool)

Setzer is a single local Go binary — a schema-blind compositor — that serves a static site together with the site's own in-site editor on localhost, and on save commits the content change to the site's Git repository, which a static host (e.g. GitHub Pages) then publishes.

Context

Setzer originates from the Wiener Bluut site: a bespoke single-page static site whose non-technical editor needs to update structured content (events, programme, testimonials, contact) with changes that actually persist and publish — without a cloud CMS. Exploring that need produced a set of constraints that no off-the-shelf tool satisfies together, so we extract the generic part into its own reusable tool.

Established constraints:

  • The site is static; Git is the content store (free versioning, rollback, diff).
  • The editor is bespoke and in-site — schema- and design-aware, edits the real page.
  • No cloud runtime and no cloud secrets. The only non-static action is the authenticated write.
  • Reusable across sites. Each site is independent; the tool is generic.
  • The editor must be runnable by anyone who can use a computer (no Git account required).

Decision

The only part of the system that cannot be static is the authenticated write. Setzer performs it as a local process running as the editor — not a deployed cloud function — which dissolves all cloud hosting and secret-management. Everything else (storage, versioning, serving) is Git + a static host.

  1. Git as storage. Content is committed (e.g. content.json) to the site repo — the single source of truth. Versioning, rollback and diff come for free.
  2. Static serving via the site's host. The public pulls a static content.json from the CDN and renders client-side. Serving needs no runtime; "build" is trivial for a vanilla site.
  3. The write runs locally. The authenticated commit is done by the local Setzer binary, acting as the editor — eliminating any deployed function, its host, and its secret store.
  4. Bespoke editor lives in the site; Setzer is schema-blind. The editor (schema + design coupled) ships with the site. Setzer never inspects content. The seam is one opaque POST /__save of the content document.
  5. Repo separation. Setzer is its own (public) repo; each site is its own repo. The only coupling is a thin convention — "a Git repo laid out the way Setzer expects" — so one tool serves N sites.
  6. Auth = GitHub PAT, in the OS keychain. A fine-grained token (contents:write on the one repo), entered via Setzer's browser admin UI and stored in the keychain. Chosen so editors with no GitHub account can still publish. later ambient-gh support; OAuth device flow.
  7. Local working clone. Setzer keeps a clone and commits/pushes from it — enabling future build steps and offline editing.
  8. Two distinct browser UIs. Setzer's own config/admin UI (which repo, paste PAT) is generic; the content editor is the site's bespoke UI. Same browser, different owners.
  9. Security by capability, not obscurity. The save endpoint binds to 127.0.0.1 only, with a one-time nonce + Origin checks (CSRF / DNS-rebinding defense). The publicly-shipped editor is inert — with no local Setzer to POST to, it cannot persist.
  10. Editor ships in public output, inert — no stripping. Discoverable but unusable for public writes by design. The only "cost" is a little JS and a visible schema, both immaterial.

MVP scope (v0)

In v0 build now

  • Browser admin UI: configure repo, paste PAT → keychain.
  • Maintain a working clone of the site repo.
  • Serve the clone on 127.0.0.1:port (loopback only).
  • POST /__save → write content.json → commit → push (PAT).
  • Optimistic safety via non-fast-forward push rejection.
  • Pure-Go / no-CGO to keep cross-compile trivial.

Deferred later

  • Ambient-gh auth support (alongside PAT).
  • OAuth device flow (for editors who do have GitHub).
  • Explicit optimistic lock via SHA / If-Match.
  • Build-step hook (run a generator before commit).
  • Multi-site management in one Setzer.
  • Packaging: Homebrew tap, Scoop/winget.

Consequences

Gains

  • No cloud runtime, no cloud secrets, all-GitHub.
  • Content versioned in Git; rollback for free.
  • One generic tool, reusable across sites.
  • Simple ops; nothing to deploy or keep alive.

Costs

  • The editor must run a local binary (acceptable: "anyone who can use a computer").
  • Per-machine setup; a token to provision per editor.
  • Not reachable from an arbitrary browser unless the tool is running.
